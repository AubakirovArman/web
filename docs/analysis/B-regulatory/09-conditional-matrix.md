# B9. Условная матрица «параметр → документы → проверки»

Синтез B2–B8: как параметры заявки определяют обязательные документы и проверки (регуляторная «истина»). Это целевая модель условности (с чем сверять реализацию — C).

## Базовая ось маршрутизации
```
object-type (ЛС | МИ)
  └─ procedure (регистрация | перерегистрация | изменения)
       └─ ЛС: product-type (13 типов)   |   МИ: класс риска (1/2а/2б/3) + вид номенклатуры
```

## ЛС — матрица (примеры правил из НПА)
| Условие (параметры) | Обязательные документы | Проверки |
|---|---|---|
| procedure=регистрация (всегда) | Заявление (М1.2), оплата (М1.1), ОХЛП, инструкция, макеты, модуль 3 (3.2.S+3.2.P), модуль 2 (резюме) | комплектность; согласованность ОХЛП↔инструкция↔макеты↔М3 |
| product-type=generic | + отчёт по **биоэквивалентности** (М5.3.1) **или** обоснование waiver | bioequivalence_report_check / waiver |
| product-type=original | + полные модули 4 (доклиника) и 5 (клиника) | module-level completeness |
| product-type=biosimilar | + сравнительные исследования с референтным | comparability |
| new-api-flag=да | + **мастер-файл АФС** | наличие/структура мастер-файла |
| biological/immunobiological=да | + спец. биологические данные | biological requirements |
| manufacturer-country≠ЕАЭС | + **CPP** (М1.2.1), **GMP**-сертификат | gmp/cpp_certificate_check (срок, орган) |
| sterile=да | данные о стерилизации/валидации | sterility_validation_check |
| любой | срок годности, условия хранения (М3, ОХЛП) | shelf_life/storage_consistency, document_expiry |
| procedure=перерегистрация | **модули 1,2** актуализированные + переоценка «польза–риск» (PSUR/RMP) | ls_reregistration_consistency |
| procedure=изменения | документы по типу IA/IB/II + затрагиваемые разделы (Прил. 19) | ls_variation_consistency; undocumented_variation |

## МИ — матрица (драйвер = класс риска)
| Условие | Обязательные документы (Прил. 4) | Проверки (п.27) |
|---|---|---|
| регистрация (всегда) | заявления (Прил.2,3), доказательства Общих требований (Реш.27), маркировка, эксплуатационный документ, анализ рисков, стабильность | соответствие Общим требованиям; маркировка; стабильность; анализ рисков |
| класс ↑ (2б/3) | + клинические доказательства, более полная СМК (Реш.106) | клинические испытания (Реш.29); СМК; инспекция производства |
| стерильное | + данные о стерилизации | методы/контроль остатков стерилизации |
| есть ПО | + данные о валидации ПО | верификация/валидация ПО, защита |
| есть ЛС в составе | + данные о ЛС-компоненте | безопасность/эффективность ЛС, совместимость |
| измерительная функция | + испытания типа средств измерений | соответствие |
| технические испытания | протоколы (Реш.28) | соответствие правилам испытаний |
| биологический контакт | оценка биологического действия (Реш.38) | биобезопасность материалов |

## Формальная модель (рекомендация для целевого решения)
Триггер документа/проверки = логическое выражение над параметрами:
```
required(documentX) ⇐ object-type ∧ procedure ∧ predicate(params…)
applies(checkY)      ⇐ documentX.present ∧ predicate(params…)
severity(missing)    = f(criticality из НПА)
source               = пункт НПА (трассируемость)
```
Сейчас в MVP это **текстовые `requiredWhenExpression`** (хрупко); целевое — структурированные предикаты с валидацией и тестами.

## Что сверять в Части C
- C1: поля заявки ↔ управляющие параметры этой матрицы (все ли триггеры имеют поле).
- C2: документы ↔ строки матрицы (полнота по модулям/Прил.4, по процедурам, по ЛС/МИ).
- C3: проверки ↔ методология (B7) и требования (B6).
- C4: процедуры ↔ реализованные ветви (сейчас только ЛС/регистрация).
