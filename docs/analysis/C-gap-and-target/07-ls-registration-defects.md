# C7. Что НЕ ТАК и НЕВЕРНО в ЛС/регистрации (реализованная ветка)

Только про ЛС/регистрацию (единственная работающая ветка). Не «чего не хватает», а **что сделано неверно/некорректно** — с доказательствами из данных.

---

## 🔴 Дефект 1. Неверный источник и нечитаемые ссылки на НПА
**Что:** все **174/174** правила в `source_reference` ссылаются на **национальный приказ МЗ РК № ҚР ДСМ-10** и указывают **«строка 101 / строка 103 …»** (номер строки распарсенного файла).
**Доказательство:**
- `by_line_number = 174` (все ссылки вида «…, строка N / строки N»).
- топ источников: «Приказ МЗ РК № ҚР ДСМ-10 …, строка 101/103/108…».
**Почему неверно:**
- Авторитетный источник состава досье ЛС — **Решение ЕЭК № 78** (ОТД/CTD, прил. № 1), а правила цитируют **национальный приказ**, причём по **номеру строки**, а не по пункту/разделу.
- «строка 101» — артефакт парсинга, **не является юридической ссылкой**. Трассируемость «замечание → пункт НПА» фактически сломана.
**Последствие:** эксперт не может сослаться на корректный пункт; нельзя проверить правомерность требования.

## 🔴 Дефект 2. Движок игнорирует структурированное условие — УТОЧНЕНО ПО ФАКТУ
**Уточнение:** условие НЕ потеряно. Есть два корректных представления:
- **`condition_json`** (174/174) — **исполняемый предикат** (DSL `all/any/eq/neq/in/not_empty/contains`). Пример 1.2.5: `{all:[{eq:[param-object-type,LS]},{eq:[param-procedure,registration]},{in:[param-product-type,[generic,hybrid,bioanalog]]}]}`.
- **`original_trigger_expression`** — человекочитаемое условие.
**Баг:** `server-store.buildAdminDossierDocumentType` строит `requiredWhenExpression` из **`condition_text`** (= описание проверки «Профиль проверки содержимого…»), а **`condition_json` в `DocumentType` не пробрасывается и не вычисляется**. Движок идёт хрупким текстовым путём `matchesLsRequirementTriggerExpression`.
**Последствие:** условность держится на случайных совпадениях слов в описании, хотя рядом есть корректный машинный предикат.
**Фикс (приоритет 1 — делается):** пробросить предикат из `condition_json` в `DocumentType` и исполнять его; текстовый путь — fallback.

## 🔴 Дефект 3. Псевдо-условность через раздутые linked_params
**Что:** `linked_params` содержит до **84 полей** на одно правило (несколько правил — 20–30).
**Доказательство:** распределение: 84×1, 30×1, 28×1, 25×2, 24, 23, 22, 21, 20×3…; правило 1.2 связано почти со всеми полями формы.
**Почему неверно:** `linked_params` = авто-сгенерированный шум (метаданные), НЕ реальное условие. Реальная зависимость — в `condition_json` (Дефект 2), который компактен и корректен. Т.е. `linked_params` вводит в заблуждение, но не используется как условие.
**Последствие:** мусорные метаданные; при переходе на `condition_json` (Дефект 2) `linked_params` можно пересобрать из предиката или удалить.

## 🟡 Дефект 4. Дубликат doc_code (коллизия)
**Что:** код **1.2.1** назначен **двум разным документам** одновременно: «Сертификат на фармацевтический продукт (CPP)» и «Сертификат GMP».
**Доказательство:** `doc_code 1.2.1 → count 2`.
**Почему неверно:** один код = два документа → при дедупликации/сопоставлении файлов один может «затереть» другой или попасть не туда.
**Последствие:** риск потери одного из требований (CPP или GMP) в чеклисте.

## 🟡 Дефект 5. Документы, которые не могут сработать (битые поля)
**Что:** **6** правил с `applicability='needs_new_param'` ссылаются на **несуществующие поля формы**.
**Доказательство:** 1.2.2 (патенты), 1.3.2, 1.5.1, 2.3.A.3 (новые вспомогательные), 3.2.P.4.5, 3.2.P.4.6 — напр. требуют `param-license-or-patent-active`, которого нет в 81 поле.
**Почему неверно:** условие ссылается на параметр, который заявитель не может заполнить.
**Последствие:** эти документы (патенты, новые вспомогательные вещества) **никогда не будут корректно затребованы**.

## 🟡 Дефект 6. Требования — обезличенный вывод LLM
**Что:** 302 требования имеют только **2 уникальных `checkType`**: «содержательная проверка Gemma» (250) и «Gemma: требования к CTD-разделу» (52).
**Почему неверно:** поле `checkType` не несёт информации (один и тот же ярлык на сотни требований); требования сгенерированы ИИ и не типизированы по сути проверки. (Слово «Gemma» в UI уже скрыли, но данные остались обезличенными.)
**Последствие:** нельзя группировать/маршрутизировать требования по типу проверки; качество требований не верифицировано против первоисточника.

## 🟡 Дефект 7. «Проверки внутри документа» не исполняются детерминированно
**Что:** `validation_checks` — это **текстовые инструкции** («Проверить наличие…», «Проверить читаемость суммы/даты/плательщика»), а не исполняемые правила.
**Почему важно:** детерминированный движок (реестр из 26 проверок) их не запускает; они идут как промпт в LLM (npa-gemma). То есть «проверка содержимого документа» для ЛС/регистрации фактически держится на LLM (со всеми рисками извлечения/качества).
**Последствие:** нет гарантированной детерминированной проверки содержимого; результат зависит от LLM и качества OCR.

## 🟡 Дефект 8. Двойная логика проверок (клиент vs сервер)
**Что:** `runPreCheck` исполняется и в wizard (клиент), и в кабинете эксперта (сервер `/check`).
**Последствие:** для одной и той же заявки заявитель и эксперт могут увидеть **разные** наборы находок.

## 🟢 Дефект 9. Метаданные нормализации не финализированы
**Что:** `normalization_status='normalized'` = **0** правил (все в другом статусе).
**Последствие:** сигнал, что правила — промежуточный (ненормализованный) результат, не приведённый к финальному виду.

---

## Корень проблем
Правила ЛС/регистрации **сгенерированы автоматически (парсинг национального приказа + LLM)** и **не выверены против первоисточника (Решение 78 ЕАЭС)**:
- источник и ссылки — нац. приказ по номерам строк;
- условие появления документа отсутствует как предикат (в его поле лежит описание проверки);
- зависимости (linked_params) раздуты;
- требования обезличены LLM;
- есть дубликаты кодов и битые условия.

## Что исправить (приоритет, в рамках ЛС/регистрации)
1. **Переписать источник/условия по Решению 78:** для каждого документа — реальный пункт НПА + **структурированный предикат** обязательности (по `procedure`, `product-type`, флагам), отдельно от описания проверки.
2. **Развести поля:** `triggerCondition` (когда нужен) ≠ `validationChecks` (что проверять внутри) ≠ `sourceReference` (пункт НПА).
3. **Сжать linked_params** до реально влияющих полей; устранить шум.
4. **Устранить дубликат 1.2.1** (разделить CPP и GMP на разные коды).
5. **Закрыть 6 needs_new_param:** добавить недостающие поля или убрать битые условия.
6. **Типизировать требования** (реальные checkType вместо «Gemma…»); сверить выборку с Решением 78/88.
7. **Свести проверки к одному движку** (сервер), убрать клиент/сервер-расхождение.
8. **Решить статус validation_checks:** какие переводим в детерминированные проверки, какие оставляем LLM (с верификацией экспертом).

> Важно: внешне демо «работает» (e2e ideal проходит), но это маскирует то, что **условность и трассируемость к НПА недостоверны** — а именно они и есть суть продукта.
