# C8. Полный аудит скриптов-проверок (ЛС/регистрация)

Ручная сверка реализаций всех раннеров: `lib/checks/runners/engine-general-runner.ts`, `runners/ls/{consistency,manufacturing,quality-support,bioequivalence,reregistration}-checks.ts`, `engine-file-helpers.ts`, `engine-utils.ts`.

## Как устроено (контекст)
- Находки создаются через `createFinding(...)`, который **НЕ ставит `checkerId`** → большинство находок привязаны к **категории** (напр. «Расхождения между документами»), а не к id каталога. Поэтому «реально эмитится» лишь ~6 `checkerId`, хотя проверок в коде гораздо больше — **они работают, просто без id**.
- Все content-проверки используют `extract(file, 'поле')` — т.е. зависят от того, что парсер/LLM извлёк это поле. Если поле не извлечено → проверка **молча пропускается** (находки нет).

### 🔴 Найденные ошибки/недоработки
1. **Баг размера шрифта** (`checkDocxFormatting`, `engine-file-helpers.ts`):
   `if (bodySizes.some((s) => s !== 12))` — флагует, если **любой** размер ≠ 12пт. В норме есть заголовки 14пт, сноски 10пт → **ложные срабатывания почти на каждом DOCX**. Нужно проверять **доминирующий/основной** размер, а не «любой ≠ 12».
2. **Заглушка «текст маркировки ↔ макет»** (`consistency-checks.ts`, п.8, строки ~244-251): `// placeholder check` — ничего не сверяет. (Сверка маркировки с *заявлением* реализована в quality-support, а вот с *макетом* — нет.)
3. **Молчаливый пропуск при пустом тексте** — `checkRequiredSections` и большинство content-проверок: `if (!text) return;`. При сбое OCR/извлечения **не создаётся находка «не смогли проверить»**, документ проходит. Маскирует проблемы извлечения.
4. **Keyword-поиск разделов** (`SPC_REQUIRED_SECTIONS`) — подстроки: иное словесное оформление раздела → ложное «отсутствует»; ключевое слово вне раздела → ложное «есть». Хрупко.
5. **`checkerId` не присваивается** `createFinding` → находки не трассируются к каталогу из 26 проверок (только ~6 имеют id). Трассируемость/аналитика по типам проверок страдает.

### 🟡 Системные ограничения (by design, но важно знать)
- Надёжность ВСЕХ content-проверок = надёжность извлечения (`extract`). Поля типа `validUntil`, `shelfLife`, `manufacturer`, `address` должны быть извлечены парсером/LLM; иначе проверка пропускается.
- Детерминированные проверки покрывают комплектность/форматы/расхождения/сроки/оформление; смысловые требования по пунктам НПА — через LLM (`npa-gemma-check`), отдельно.

## Общий вердикт
Рукописные скрипты-проверки в целом **корректны и хорошо нацелены**: правильный гейтинг (по типу препарата, флагам, наличию файлов), аккуратные ссылки на конкретные пункты НПА (Решение 78/85/88, Приказ ДСМ-10). Это качественно лучше, чем авто-сгенерированные `validation_checks` правил.
**К исправлению:** (1) баг размера шрифта, (2) заглушка маркировка↔макет, (3) добавить находку «не удалось проверить» при пустом извлечении, (4) опц. — присваивать `checkerId` для трассируемости.
