# C5. Требования к целевому решению

Синтез Частей A, B и gap-анализа (C1–C4). Цель: автоматизация для заявителя и эксперта — экономия времени, заблаговременный показ ошибок, привязка замечаний к НПА.

## Функциональные требования

### F1. Динамическая форма заявки (заявитель)
- Поддержка **всех веток** {ЛС, МИ} × {регистрация, перерегистрация, изменения}.
- **Условная видимость полей** по единым предикатам (процедура, тип препарата ЛС, класс риска МИ) — синхронно с движком документов.
- Поля форм НПА 1:1 (ЛС Формы I–III; МИ Прил. 2,3), включая блок «Вносимые изменения» (IA/IB/II) и драйвер «класс риска».
- Трассируемость: поле ↔ пункт формы НПА.

### F2. Движок обязательных документов
- Единая модель правил для всех {домен × процедура}; условия — **структурированные предикаты** (не текст).
- Состав по CTD (ЛС) и Приложению № 4 (МИ); условность по типу/классу.
- Автосопоставление загруженных файлов с разделами досье (в т.ч. загрузка папкой).

### F3. Движок проверок (предэкспертиза)
- Детерминированные: комплектность, форматы, OCR, расхождения значений, сроки сертификатов, наличие разделов.
- Смысловые (LLM): соответствие требованиям пунктов НПА (ОХЛП/Решение 88; Общие требования/Решение 27).
- **Чек-лист МИ п.27** (а–м) как набор проверок.
- Каждая находка: severity, рекомендация, **ссылка на пункт НПА**, доказательство (цитата/страница/файл).

### F4. Кабинет эксперта
- Предварительный протокол: находки сгруппированы по документам/разделам, статусы (принять/отклонить/ложноположит./нужны уточнения).
- Формирование запроса заявителю из находок.
- Этапность процедуры (сроки, статусы) согласно B2/B3.

### F5. Справочник НПА
- Единый источник: документы + секции + полнотекстовый поиск + ИИ-анализ.
- Привязка требований/проверок к пунктам.

### F6. Администрирование
- Управление: поля, типы документов, правила (предикаты), требования, проверки, НПА.
- Версионирование правил и НПА; актуальные редакции (учёт «утратил силу»).

## Нефункциональные требования
- **NF1. Данные:** нормализованная модель (заявки, файлы, находки, результаты проверок — отдельные таблицы), а не монолитный JSONB.
- **NF2. Производительность:** списки-сводки, пагинация, кэш справочника/конфига (уже есть в MVP).
- **NF3. Надёжность LLM:** очередь задач, ретраи, кэш результатов по (хэш файла, требование), деградация без падения.
- **NF4. Безопасность/роли:** аутентификация и разграничение (заявитель/эксперт/админ).
- **NF5. Аудит/трассируемость:** кто/когда менял правила и статусы находок; ссылки на НПА везде.
- **NF6. Доступность/UX:** мобильная навигация, обе темы, единые состояния (skeleton/empty/error), доступность.
- **NF7. Тестируемость:** юнит-тесты движков правил/проверок + e2e по веткам процедур.
- **NF8. Сопровождаемость:** разделение reference-БД и runtime-БД; удаление backup-таблиц и legacy-роутов.

## Принципы автоматизации (граница ответственности)
- Система — **рекомендательная предэкспертиза**: показывает ошибки/риски, не выносит решение.
- Чёткое разделение: авто-детерминированное / LLM-подсказка (с проверкой экспертом) / только эксперт (научная оценка, «польза–риск», классификация риска).

## Ключевые открытые вопросы (решить с владельцем)
1. Паритет ЛС/МИ сразу или поэтапно (сначала добить ЛС: перерег+изменения, затем МИ).
2. Нужен ли **национальный слой РК** (приказы МЗ: приём, экспертные организации, ценообразование) или только ЕАЭС-уровень.
3. Глубина LLM-автоматизации (какие проверки доверяем ИИ, какие — только подсказка).
4. Управление редакциями НПА (как поддерживать актуальность).
