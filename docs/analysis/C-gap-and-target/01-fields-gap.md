# C1. Покрытие полей: реализация (A10) ↔ регуляция (B4)

## Итог
В MVP **81 поле** для ЛС/регистрации (Форма I). Покрытие Формы I ЛС — **хорошее**. Пробелы — в формах перерег/изменений ЛС и в форме МИ.

## Покрыто хорошо (ЛС, Форма I — регистрация)
- Идентификация: торг. наименования (kz/ru/en), МНН, АТХ, ЛФ, дозировка, состав, упаковка, способы введения.
- Тип препарата (`product-type`, 13 значений) = типы Решения 78 ✅.
- Производство: производитель/площадки/QC-лаборатория/трансфер.
- Характеристики: срок годности, хранение, транспортировка, периоды после вскрытия/растворения, стерильность.
- Статусы: биологический/иммунобиологический, новая АФС, орфанный (блок), ВОЗ-преквалификация, доп. мониторинг, биоэквивалентность, клинические исследования.
- Заявитель/оплата/референтный препарат.

## Условная видимость
- В MVP видимость полей — клиентские группы под-шагов (`parameter-groups.ts`).
- В регуляции видимость определяется выбором процедуры и типа препарата/класса риска.
- **Расхождение:** нужно сверить, что каждое условное поле НПА появляется по тому же триггеру (формальная модель B9).

## Рекомендации
1. Достроить **поля процедуры подачи** (взаимное признание/децентрализ./приведение) для ЛС.
2. Реализовать **формы перерег и изменений** ЛС (поля уже частично есть).
3. Добавить **форму МИ** с драйвером «класс риска» + номенклатура.
4. Перевести условную видимость на **единые предикаты** (общие с движком документов, B9), а не клиентские группы.
5. Сверка поле-в-поле с формами НПА — оформить как таблицу соответствия (трассируемость поле ↔ пункт формы).
