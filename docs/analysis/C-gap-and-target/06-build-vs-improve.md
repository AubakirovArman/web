# C6. Доработка vs переписывание + целевая архитектура

## Вывод (рекомендация)
**Дорабатывать текущий MVP эволюционно, а не переписывать с нуля.**

Обоснование: ядро ценности уже есть и работоспособно — слоистая архитектура (тонкие роуты, логика в `lib/`), 174 нормализованных правила ЛС с привязкой к НПА, справочник (42 909 секций) + ИИ-анализ, микросервисы извлечения/LLM, дизайн-система, оптимизированная производительность. Переписывание выбросит этот актив. Пробелы (МИ, перерег/изменения, нормализация данных, единый движок) закрываются **рефакторингом и наполнением**, а не сменой платформы.

> Исключение: если появится требование принципиально иной платформы/интеграции (гос. ИС, иной стек) — тогда переписывание; на текущих данных оснований нет.

## Что переиспользовать как есть
- Next.js-каркас, дизайн-система (токены, обе темы, бейджи).
- Микросервисы document-parser (8051) и gemma-checker (8052) + интеграция vLLM.
- Справочник (`reference_*`, experiment) — объединить в один источник.
- 174 правила ЛС/регистрации, реестр проверок, профили форм.

## Что рефакторить (технический фундамент)
1. **Модель данных:** вынести из JSONB в таблицы — `applications`, `application_files`, `findings`, `check_results` (нормализация, аналитика, масштаб).
2. **Единый движок правил:** одна схема `requirement_rules` со scope = object-type × procedure × (тип/класс); условия — **структурированные предикаты** + валидация + тесты (вместо текстовых `requiredWhenExpression`).
3. **Единый движок проверок:** убрать дублирование клиент/сервер; проверки на сервере, клиент — отображение.
4. **LLM-слой:** очередь + кэш по (хэш файла, требование) + ретраи.
5. **Справочник:** один источник (документы+секции+полнотекст+анализ); удалить legacy `/api/reference`, backup-таблицы; развести reference/runtime БД.
6. **Auth и роли.**
7. **UX-долги** (мобильное меню, metadata, AlertDialog, формы на react-hook-form/zod).

## Что достроить (наполнение по приоритету)
1. **ЛС/изменения** (есть `ls_variation_checklist_rules`) — быстрый выигрыш.
2. **ЛС/перерегистрация** (модули 1–2 + PSUR).
3. **МИ/регистрация** — полноценный второй домен: форма (класс риска), документы (Прил. 4), чек-лист п.27.
4. МИ/перерег и изменения.

## Целевая архитектура (эскиз)
```
Frontend (Next.js, роли)
  ├─ Заявитель: динамическая форма (предикаты) → состав документов → загрузка → предпроверка
  ├─ Эксперт: протокол находок (по НПА), статусы, запросы, этапность
  └─ Админ: правила/проверки/НПА/поля (версионирование)
Backend (API + сервисы)
  ├─ Rules Engine (predicates)         ← requirement_rules (норм. БД)
  ├─ Checks Engine (детерминир.)       ← реестр проверок
  ├─ LLM Orchestrator (очередь+кэш)    → gemma-checker → vLLM
  ├─ Extraction                        → document-parser
  └─ Reference (документы+секции+поиск+анализ)
БД (PostgreSQL, нормализованная):
  applications, application_files, findings, check_results,
  requirement_rules, parameters, document_types, reference_documents/sections,
  npa_requirements, users/roles, audit_log
```

## Дорожная карта (этапы)
- **Этап 0 — фундамент (рефакторинг):** нормализация данных, единый движок правил (предикаты), auth, UX-долги.
- **Этап 1 — добить ЛС:** изменения + перерегистрация (формы, правила, проверки, тесты).
- **Этап 2 — МИ:** регистрация (класс риска, Прил. 4, п.27), затем перерег/изменения.
- **Этап 3 — качество ИИ:** кэш/очередь, сверка ОХЛП(№88)↔проверки, метрики качества.
- **Этап 4 (опц.) — национальный слой РК** (если нужен).

## Критерии успеха
- Каждая ветка {домен×процедура} даёт корректный список документов и предпроверку.
- Каждая находка трассируется к пункту НПА.
- Эксперт получает готовый протокол; заявитель видит ошибки до подачи.
- Покрытие тестами движков и веток процедур.
