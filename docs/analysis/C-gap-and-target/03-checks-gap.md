# C3. Покрытие проверок: реализация (A6/A7) ↔ методология (B6/B7)

## Итог
MVP: **26 проверок** в реестре + LLM-проверка требований (npa-gemma). Хорошо покрыты технические и согласованность; смысловые — через LLM; научная оценка остаётся эксперту. Главные пробелы — методология **МИ (п.27)** и глубина требований **ОХЛП (Решение 88)**.

## Требования по пунктам (B6 — Решение 88, ОХЛП)
- 302 требования в реестре + 174 правила с `validation_checks`.
- **Открыто:** полнота/точность переноса ОХЛП (250 секций Решения 88) в проверки. Рекомендуется выборочная сверка: 10 пунктов ОХЛП ↔ соответствующие `validation_checks`.

## Качество LLM-проверки
- Есть отчёт `docs/npa-gemma-quality-report.md` — учесть метрики.
- Риски: зависимость от извлечения/OCR; нет кэша по (файл, требование); внешний vLLM.

## Что автоматизируемо vs остаётся эксперту
- **Авто (детерминир.):** комплектность, форматы, OCR, расхождения значений, сроки сертификатов, наличие разделов.
- **Авто (LLM, с проверкой эксперта):** соответствие текста требованию пункта НПА.
- **Эксперт:** научная оценка качества/безопасности/эффективности, «польза–риск», валидность доказательств, классификация риска МИ.

## Рекомендации
1. Реализовать **чек-лист МИ п.27** как набор проверок (а–м) с драйвером класса риска.
2. Достроить/проверить «заглушечные» проверки (15–26): module3, sterility, pharmacovigilance, variation.
3. Выборочно сверить ОХЛП (Решение 88) ↔ `validation_checks`; докрыть пробелы.
4. Свести **двойную логику** (клиент/сервер) к единому движку проверок.
5. Ввести **кэш результатов LLM** по (хэш файла, id требования) + ретраи/очередь.
6. Каждая находка → ссылка на пункт НПА (трассируемость, уже частично есть).
