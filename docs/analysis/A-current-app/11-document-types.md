# A11. Типы документов / правила (document_requirement_rules)

Источник: таблица `document_requirement_rules` (174 активных) + admin `/api/admin/document-types`.
Каждая строка = «документ/раздел досье + условие обязательности + проверки + ссылка на НПА».

## Покрытие
- **Все 174 строки = scope `LS / registration`.** ❗ Нет МИ. Нет перерегистрации. Нет внесения изменений.
- `row_type`: **document — 174**, section — 0 (по факту строки трактуются как документы/разделы).
- Полнота полей: **validation_checks — 174/174**, **linked_params — 174/174**, **source_reference — 174/174** (каждое правило имеет проверки, привязанные поля и пункт НПА).

## Условие обязательности (applicability)
| Значение | Строк | Смысл |
|---|---|---|
| `conditional_required` | 94 | обязателен при условии (зависит от `linked_params`) |
| `always_required` | 74 | всегда обязателен |
| `needs_new_param` | 6 | требует параметр, которого пока нет в форме (напр. `param-license-or-patent-active`) |

## Модель условности
- `linked_params` (jsonb) — массив id полей заявки, влияющих на правило.
- `condition_text` / `show_logic` / `condition_json` / `original_trigger_expression` — выражение появления документа.
- `validation_checks` (jsonb) — текстовые проверки внутри документа.
- `source_reference` — пункт НПА (источник требования).
- `normalization_status` / `confidence` — метаданные нормализации (правила извлекались/нормализовались, в т.ч. ИИ).

**Пример (1.2.5 «Декларация по ОХЛП»):** applicability=`conditional_required`; linked_params включают bioequivalence-required, biological/immunobiological, clinical-studies, new-api-flag, reference-product, shelf-life, storage-conditions, trade-name-ru, inn-ru — т.е. появление декларации завязано на тип/характеристики препарата.

## Как используется в рантайме
- Admin: постранично через `/api/admin/document-types` (серверная пагинация, 174 → страницы).
- Wizard/Expert: на чтении `admin_runtime_config` подменяет `documentTypes`/`lsDossierDocumentTypes` данными из этой таблицы (кэш 2 мин).
- Движок правил (`getRequiredDocuments`) использует это для построения чеклиста (детали в A5, этап 2).

## Наблюдения / gap-сигналы (для Части C)
- **Главный пробел: только ЛС/регистрация.** Для паритета нужны наборы: ЛС/перерег, ЛС/изменения, МИ/регистрация, МИ/перерег, МИ/изменения (источники — Решение 78 разд. VIII/IX, Решение 46 + Приложение 4).
- 6 правил `needs_new_param` — требуют доопределить поля в форме.
- Правила нормализованы (в т.ч. ИИ) — нужно сверить с первоисточником (B5/B6, gap C2): полнота перечня документов модулей (Решение 78, «Перечень документов в модулях», стр. 4314 md).
- `validation_checks` — текстовые; как они исполняются движком — разобрать в A6 (этап 2).
