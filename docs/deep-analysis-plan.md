# План глубокого анализа: автоматизация экспертизы ЛС/МИ

> **Цель проекта.** Автоматизация для эксперта и заявителя: экономить время, заранее
> показывать ошибки/несоответствия, привязывать замечания к пунктам НПА.
> Текущее веб-приложение — **MVP**. Прежде чем строить новое решение (или дорабатывать текущее),
> нужно собрать **полную** информацию: (A) что уже сделано, (B) что говорит нормативка ЛС/МИ,
> (C) где расхождения и каким должно быть целевое решение.
>
> **Важно:** на этапе анализа ничего в приложении не создаём и не меняем — только изучаем и документируем.

## Принципы анализа
- Каждый пункт плана даёт **отдельный артефакт-файл** в `docs/analysis/`.
- Для каждого аспекта фиксируем: **источник** (код/БД/НПА), **метод** (чтение/grep/SQL), **вывод**, **открытые вопросы**.
- Нормативка — «истина»; реализация сверяется с ней (gap-анализ в Части C).
- Везде, где возможно, ссылки: на файл+строку (код) и на пункт/раздел НПА (регуляция).

## Структура выходных артефактов
```
docs/analysis/
  A-current-app/        # что сделано (MVP)
  B-regulatory/         # методология ЛС/МИ
  C-gap-and-target/     # сопоставление и требования к целевому решению
  INDEX.md              # навигация по всем артефактам + статус
```

# Порядок и приоритеты выполнения

**Этап 1 (фундамент) —** A1, A2, A3, A10, A11 + B1.
*Зачем:* без модели данных, полей и карты документов остальное не на чём строить.

**Этап 2 (логика MVP) —** A4, A5, A6, A7, A8, A9, A12, A13.

**Этап 3 (регуляторная истина) —** B2, B3, B4, B5 → затем самые объёмные B6, B7, B8, B9.

**Этап 4 (синтез) —** C1–C6.

> Самые трудоёмкие: **B6** (требования по пунктам, особенно Решение 88 — 250 разделов) и **B9** (условная матрица). Их делать после того, как готовы A10/A11/B4/B5.

# Методы и инструменты сбора
- **Код:** чтение модулей, построение карт зависимостей (импорты), фиксация файл:строка.
- **БД:** SQL по `reference_sections` (структурно), `document_requirement_rules`, `runtime_dictionaries`.
- **НПА:** структура через БД (heading_number/full_heading), полный текст через `md_извлечено/`.
- **UI:** прогон страниц на проде (8040) для фиксации фактического поведения.
- **Сверка:** двусторонняя трассируемость регуляция↔код (таблицы соответствия).

# Критерии готовности анализа
- [ ] Каждое поле заявки имеет источник в НПА (или помечено как «доп./служебное»).
- [ ] Каждый тип документа сопоставлен с модулем CTD / Приложением 4 и условием обязательности.
- [ ] Каждая авто-проверка сопоставлена с пунктом методологии эксперта.
- [ ] Все 3 процедуры ЛС и процедуры МИ описаны и сопоставлены с реализацией.
- [ ] Сформированы требования к целевому решению и решение «доработка vs переписывание».
