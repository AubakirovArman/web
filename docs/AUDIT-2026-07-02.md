## 🔴 КРИТИЧНО

### 1. IDOR: любой залогиненный видит и меняет ЧУЖИЕ заявки
**Где:** `lib/auth/permissions.ts:27` (в `GATES` нет `/api/applications` → `requiredPermissionFor` = `null` →
middleware пропускает при наличии любой сессии); `lib/applications/server-store.ts:79`
(`SELECT ... FROM runtime_applications` без `WHERE`); под-ресурсы `app/api/applications/[id]/findings/[findingId]/route.ts`,
`.../[id]/check/route.ts`, `.../[id]/test-submit/route.ts` не проверяют владельца.
**Риск:** заявитель читает/удаляет чужие досье и искажает результаты экспертизы. Для системы с
конфиденциальными регистрационными досье — критично.
**Как чинить:**
- Скоупить чтение/запись по владельцу: `applicant` → только `applicant_user_id === session.sub`;
  `expert` → `assigned_expert_user_id` (или по праву expert); `admin` → всё.
- Добавить `WHERE applicant_user_id=$/assigned_expert_user_id=$` в `readApplicationById` и
  `readApplicationsFromPostgres` (колонки и индексы уже есть, `runtime-postgres.ts:117-129`).
- `DELETE/PUT/PATCH` предварять проверкой владения → `403/404`.
- Добавить гейт для `/api/applications` в `permissions.ts`, чтобы middleware не пропускал по одной сессии.
**Оценка:** 1–2 дня.

## 🟠 ВЫСОКИЙ ПРИОРИТЕТ

### 3. Запись = перезапись ВСЕЙ таблицы заявок (lost update)
**Где:** `lib/applications/server-store.ts:100-162` — каждое изменение делает
`DELETE WHERE NOT id=ANY(...)` + `UPSERT` всех строк; `upsertApplication`/`patchApplication` (`:164-196`)
читают весь список ради одной строки; очередь `enqueueWrite` (`:5-14`) — в памяти одного процесса.
**Подтверждение:** в живой БД у всех 7 заявок идентичный `updated_at`.
**Риск:** потеря данных при втором инстансе/рестарте; стоимость записи растёт как O(n).
**Как чинить:** один `INSERT ... ON CONFLICT (id) DO UPDATE` по одной строке (SQL уже написан внутри
`writeApplicationsToPostgres`); `patchApplication` = `readApplicationById(id)` + точечный UPDATE;
`DELETE` — только явной операцией удаления. **Оценка:** часы.

### 4. Конфиг-блоб 6.1 MB, ~5 MB — мёртвые дубли
**Где:** `admin_runtime_config` key='default' хранит `documentTypes` (3.0 MB) и `lsDossierDocumentTypes`
(2.1 MB), которые при чтении всё равно подменяются выборкой из `document_requirement_rules`
(`lib/admin/server-store.ts:759-767`). Каждый клик привязки НПА переписывает весь блоб без защиты от гонок.
**Как чинить:** в `writeAdminRuntimeConfigToPostgres` (`server-store.ts:1126-1141`) перед сериализацией
обнулять `documentTypes` и `lsDossierDocumentTypes`. **Оценка:** часы.

### 5. Нет механизма миграций → свежий деплой сломает авторизацию
**Где:** схема через `CREATE IF NOT EXISTS` в рантайме (`runtime-postgres.ts:89-96`) создаёт `app_users`
**без** `username`/`password_hash`; эти колонки добавляются отдельным ручным SQL
(`db-scripts/001-users-and-credentials.sql`), а код авторизации (`lib/auth/users.ts`) на них жёстко завязан.
**Как чинить:** каталог `migrations/*.sql` с номерами + таблица `schema_migrations` + ~30-строчный раннер
на `pg` (или `node-pg-migrate`). Первая миграция — baseline текущей ЖИВОЙ схемы (вкл. `app_users.username`,
`password_hash`). **Оценка:** дни.

### 6. PostgreSQL :55440 не переживёт ребут

**Как чинить:** system-юнит (`User=arman`, `ExecStart=/usr/bin/postgres -D <datadir> -p 55440 -h 127.0.0.1`,
`Restart=always`, `enable`), затем корректно погасить ручной процесс.
> ✅ Уточнение: Python-сервисы 8050/8051/8052 — наоборот, полноценные systemd-юниты, enabled, переживут ребут.

## 🟡 СРЕДНЕЕ / НИЗКОЕ
- **Stored XSS через загрузку:** `text/html` проходит MIME-проверку (условие отклонения по И, а токен
  `'text'` — подстрока `text/html`) и отдаётся `inline` (`lib/api/upload-guard.ts`,
  `app/api/files/[fileId]/route.ts:44-51`). Фикс: отклонять по ИЛИ (нужны И расширение, И MIME), убрать
  широкий токен `'text'`, не доверять `octet-stream`; форсить безопасный `content-type` при отдаче.
- **`test-submit` всегда падает 401 в проде:** внутренний self-fetch на `127.0.0.1:8040` без cookie сессии
  (`app/api/applications/[id]/test-submit/route.ts:46-54`) → middleware отдаёт 401. Фикс: вызывать доменные
  функции напрямую или пробрасывать сервисный токен.
- **Куки `secure:false`** (`app/api/auth/login/route.ts:29`) — ок для HTTP-localhost, но должно быть
  env-gated (`AUTH_COOKIE_SECURE`) под HTTPS. Middleware не вычищает клиентские `x-user-*` на public-путях.
- **Нет rate-limit на логин** + энумерация пользователей по таймингу (scrypt только для существующих юзеров).
- **Разнобой формата ошибок** в API; нет zod-валидации тел на write-роутах (тихая порча конфига, 500 из
  глубины store). Роуты-сироты (`/api/reference`, `npa/add-requirements`, `npa/bind-bulk`).
