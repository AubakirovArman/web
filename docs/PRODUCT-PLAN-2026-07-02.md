**Что это.** Разбор всего приложения под ОДНУ цель: *дать экспертам инструмент, чтобы они сами
(без программиста) настраивали логику — типы документов, разделы, требования с обоснованием по НПА,
условия применимости — и чтобы заявки проверялись по этой логике.*
**Метод.** 9 агентов прочитали реальный код каждой страницы + настоящие НПА (`md_извлечено/ЛС`,
`md_извлечено/МИ`, Решение №78, рег.досье ЛС). 18 ключевых утверждений прошли адверсарную проверку —
**все подтверждены**.

**Главный вывод одной фразой:** обещание «эксперт всё настраивает сам» выполнено примерно наполовину.
Для *ЛС/регистрация* эксперт реально может менять типы документов, требования и условия — и правки
доходят до проверки. Но: (1) конструктор условий инженерно хорош, но **не выражает реальную логику НПА**;
(2) ключевые действия self-service **недоступны или молча не сохраняются**; (3) МИ/перерегистрация/изменения —
**пустые вкладки**; (4) работа эксперта **не завершается заключением**.

## 🔴 Сквозные блокеры (важнее всего — бьют по главной цели)

1. **Решения эксперта ни на что не влияют.** «Согласен/Не согласен» сохраняется в
   `expertCheckDecisions`, но нигде не читается: итоги, метрики, статус разделов считаются только по
   авто-статусам. → *Конечная цель «логическая проверка с выводом» не закрыта.*
   `document-review-dialog.tsx:365-390`, `useApplications.tsx:330-338`.
2. **Нет итогового заключения по заявке.** Нет экрана/объекта «вердикт (принять/на доработку/отказать)
   + перечень подтверждённых несоответствий + ссылки на НПА + экспорт». Есть только копирование запроса в
   буфер. `expert-header.tsx:65-69`.
3. **Поле заявки нельзя добавить/изменить из UI — только правкой кода.** Экран «Поля» read-only, API
   только GET, определения в статических JSON-сидах. → Новый атрибут для конструктора не появится без
   передеплоя. `app/admin/fields/page.tsx`, `api/admin/fields/route.ts:37`, `seed.ts:6-7`.
4. **МИ / перерегистрация / внесение изменений — пустые области.** В БД только `LS|registration|174`;
   селекторы Область×Процедура есть, а данных и UI-наполнения для остальных нет.
   `document-types/page.tsx:144-159`, `ls-registration-resolver.ts:154-155`.
5. **Конструктор не выражает вложенную логику** (И+ИЛИ вместе) — при любой вложенности показывает сырой
   JSON. Реальные предикаты НПА почти всегда вложенные. `condition-builder.tsx:42,95-103`.
6. **Модель не выражает «документ X ИЛИ Y»** (CPP ИЛИ рег.удостоверение; GMP ИЛИ EudraGMP) и
   **кардинальность** «на каждого производителя / на каждую АФС». `39. рег.досье ЛС.md:22,107-108,179`.

## Конструктор условий (центр вопроса: удобен ли, правильно ли сделан)

любую логику НПА» — недотягивает.** Переделать нужно и модель (DSL), и UI.

**Что менять/добавлять (по приоритету):**

  порогов: «не менее 10 лет», «5 лет и более», «в течение 3 лет». Добавить `gt/lt/gte/lte/between` в
  `ConditionLeaf` (`types.ts:97-104`), в `operatorsForType` (`condition-attributes.ts:42-43`), в движок и
  в UI. *(1-2 дня)*
- `now` **Вложенность И+ИЛИ в UI (группы).** Движок уже умеет `all/any/not` рекурсивно — редактор нет.
  Сделать дерево групп (группа = И/ИЛИ + строки/подгруппы), убрать единый глобальный join.
  `condition-builder.tsx:19,53-58`. *(3-5 дней)*
- `now` **Подключить `validateCondition` к экрану.** Функция написана (`condition-attributes.ts:119-142`),
  но **нигде не вызывается** — эксперт не видит ошибок (атрибут исчез, значение вне списка → условие молча

- `soon` **Оператор НЕ (not) в UI** — в DSL/движке/превью есть, в редакторе нет. `types.ts:109`.
- `soon` **Объектная форма `contains {param, where:{field,eq}}` из UI** — «если хоть один производитель

- `soon` **Зависимости документ-от-документа / «документ загружен»** — добавить в каталог атрибутов не
  только поля заявки, но и «наличие/значение другого документа». `condition-attributes.ts:64-82`.
- `soon` **Тест-прогон условия** на примерной заявке + предупреждение о взаимоисключающих строках

  обоснование требований», но «не трогай пользователей/структуру/условия». Сейчас `admin:npa` открывает
  всё сразу. `permissions.ts:16-21`.
- `now` **Закрыть эскалацию через `admin:roles`:** носитель может выдать себе `['*']`/`admin`
  (`roles.ts:34,46`) — нельзя раздавать права выше своих.
- `soon` **Аудит изменений логики** (кто/когда/что именно в требовании/условии) — сейчас только
  `updated_by/updated_at`, last-writer-wins. Для юридически значимой экспертизы обязательно.
- `soon` **Черновик → публикация → версии набора правил** с откатом — чтобы готовить изменения под новую
  редакцию НПА, не ломая идущие экспертизы.
- `later` Пояснения к правам в редакторе; несколько ролей на пользователя; немедленный отзыв прав (сейчас
  до перелогина).

## Выразимость модели vs реальные НПА (фундамент корректности)

Модель `тип → раздел → требование + ConditionNode (all/any/not · eq/neq/in/contains/empty)` работает для
простой булевой обязательности по scope и типу препарата. **Реальная логика НПА богаче — вот чего нет:**

| Конструкция НПА | Пример (источник) | Статус |
|---|---|---|
| Документ **X ИЛИ Y** (взаимозаменяемые) | CPP ИЛИ рег.удостоверение; GMP ИЛИ EudraGMP (рег.досье п.1.2.1) | ❌ нет |
| **Кардинальность** «на каждого производителя / каждую АФС» | IА2-IА4 на всех производителей; 3.2.S на каждую АФС | ❌ нет |
| **Числовые пороги / даты** | «не менее 10 лет», «5 лет и более», «в течение 3 лет» | ❌ нет |
| **Вложенные И+ИЛИ** | «(генерик ИЛИ биоаналог) И зарубежный И НЕ перерег» | ❌ только плоское |
| **Условные ПОДразделы внутри документа** | 3.2.P.5.2 при перерег; доп.сведения 3.2.S для животного происхождения | ❌ нет |
| **Зависимость документ↔значение** | «серия совпадает с образцом»; «не менее 3 серий»; согласованность срока годности | ❌ нет |
| **Ядро досье по процедуре** | перерег = части I,II / модули 1-3; закрытые части DMF по запросу | ⚠️ меткой, не выведено |
| **Матрица изменений IA/IAНУ/IB/II + группировка** | доп. V к прил.19; «группа IA+один IB → процедура IB» | ❌ нет |
| **Вариант структуры CTD vs отечественный** | один документ — два кода (ОХЛП: I.В.1 vs 1.3.3) | ⚠️ метка+хардкод |
| **«при наличии / при необходимости»** (мягкая обязательность) | GMP при наличии; данные пострег.опыта при наличии | ❌ нет |
| **МИ: класс риска как ось** | для 2б/3 — клин.доказательства, программа испытаний (Реш.46/106) | ❌ не наполнено |

**Вывод по модели:** нужна эволюция схемы, а не переписывание. Приоритетно добавить: (1) числовые
операторы + вложенность (уже в разделе про конструктор), (2) группу «логический документ» с альтернативами
(X ИЛИ Y) и вариантами кодов CTD/отечественный, (3) статус «мягкая обязательность (при наличии)», (4)
условность на уровне подразделов, (5) отдельную типологию для процедуры «изменения».

## Дорожная карта (что делать в каком порядке)

**Этап 1 — закрыть обманчивость и завершить цикл эксперта (дни):**
блокеры 1-2 (агрегация решений + заключение) → одна логика «Проверить» → перестать молча отбрасывать
checkIds → редактирование обоснования НПА у требования → «создать раздел» и «добавить/править требование
вручную» → подключить `validateCondition`.

**Этап 2 — сделать конструктор пригодным для реальной логики (1-2 недели):**
числовые операторы → вложенные И+ИЛИ (группы) → НЕ и объектный contains в UI → тест-прогон условия.

**Этап 3 — self-service полей и наполнение областей (недели):**
CRUD полей заявки (+ атрибуты конструктора из БД) → видимость полей через условия → наполнение
МИ/перерег/изменения (импорт + типология изменений) → альтернативы документов и мягкая обязательность в модели.

**Этап 4 — безопасное делегирование (параллельно):**
роль методолога + запрет эскалации → аудит изменений → черновик/публикация/версии правил.
